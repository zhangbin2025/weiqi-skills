架构、原理与设计哲学
生成时间：2026年3月16日
OpenClaw 的整体架构是什么？

Gateway 架构

核心：单一 WebSocket Gateway（默认 127.0.0.1:18789）作为控制平面。

接入层：20+ 消息渠道（QQ、Telegram、WhatsApp、Discord 等）统一接入 Gateway。

客户端：CLI、Web UI、macOS App、iOS/Android Nodes 通过 WS 连接。

运行时：Pi Agent（嵌入 pi-mono）处理推理和工具调用。

Pi Agent 的实现原理是什么？

运行时

基于 pi-mono 的 Agent 运行时，OpenClaw 改造为 RPC 模式：

1. agent RPC 接收任务 → 返回 runId（异步）

2. runEmbeddedPiAgent 序列化执行（每会话独立队列）

3. 事件流桥接：subscribeEmbeddedPiSession 将 Pi 事件转为 OpenClaw 流

4. 超时保护：默认 600s，工具调用独立计时

工作流如何存储？

存储

技能定义：workspace/skills/<name>/SKILL.md（Markdown + YAML）

会话历史：agents/<id>/sessions/<id>.jsonl（按事件逐行记录）

长期记忆：workspace/MEMORY.md（跨会话保留）

每日日志：workspace/memory/YYYY-MM-DD.md

大模型如何理解技能文档？

自然语言

SKILL.md 直接注入系统提示词，模型像阅读操作手册一样理解：

• 当用户询问天气时 → 识别触发条件

• 使用 web_search 搜索 → 知道调用什么工具

• 示例和格式说明 → 学习输出规范

不同于传统 Function Calling 的 JSON Schema，OpenClaw 用自然语言描述流程。

非文本操作（点击、拖拽、语音）如何处理？

文本化

核心原则：所有输入必须文本化，输出可以多媒体化。

• 浏览器：CDP accessibility tree（ref=e12 而非像素坐标）

• 视觉：截图 + 多模态模型识别 → 生成坐标

• 语音输入：STT 转文字后才进入模型

• 语音输出：TTS 生成音频文件发送

• 拖拽：封装为 drag(from, to) 工具调用

哪些大模型支持视觉理解？

多模态

商业：GPT-4o、Claude 3.5 Sonnet、Gemini 2.0、Qwen-VL、Kimi k1.5

开源：LLaVA、Qwen2-VL、InternVL2、MiniCPM-V

推荐：网页截图用 Claude 3.5，中文文档用 Qwen-VL，实时流用 Gemini。

固化工作流需要强算力模型吗？

成本优化

不需要。可按场景选择：

• 简单问答 → 轻量模型（GPT-4o-mini、Qwen-Turbo）

• 固化工作流 → 中等模型（Claude Haiku）

• 复杂推理 → 大模型（Claude 3.5、GPT-4o）

支持本地部署（Ollama），成本接近零。

如何防止工作流死循环？

安全防护

• 超时：单轮默认 600s，工具调用独立超时

• 次数限制：单轮最多 20 次工具调用

• 循环检测：A→B→A→B 模式自动拦截

• 去重：相同参数调用短期缓存（~5分钟）

• 队列模式：steer 模式下新消息打断当前执行

除 exec 外还有哪些执行方式？

工具矩阵

• web_search/fetch → HTTP API 调用

• browser → CDP 浏览器自动化

• nodes → 远程设备 RPC（iOS/Android）

• sessions_spawn → 子代理（ACP/Subagent）

• cron → 定时任务调度

• canvas → 可视化界面操作

• message → 跨渠道消息发送

创始人是谁？设计哲学是什么？

理念

Peter Steinberger（PSPDFKit 创始人）

核心：为 Molty（太空龙虾 AI）打造真正的个人助理，而非聊天机器人。

原则：

• Local-First：数据留在本地，拒绝 SaaS 订阅

• 渠道即界面：不新建 App，在你已用的渠道中工作

• 成为某人：有身份、记忆、边界，而非工具

版本历程如何？

CalVer

采用日历版本：vYYYY.M.D

• 2024.11 → 项目启动，基础 Gateway

三通道：stable / beta / dev

代码规模多大？能否保持轻量？

规模

• 源码：~5-10 万行 TypeScript

• 打包：3000+ 模块

• 运行时：Gateway ~100-200MB，对话 +50-100MB

模块化保持轻量：

• 不配 skills → 不加载

• 不配 nodes → 无设备开销

• 不配 browser → 不启动 Chrome

隐私如何保护？

安全

• 存储：全部本地（~/.openclaw/）

• 模型选择：本地 Ollama（完全离线）或信任 API（Anthropic 承诺不训练）

• 记忆：用户控制的 Markdown 文件

• 压缩：自动清理对话历史，保留摘要

长期/7x24 小时任务如何处理？

异步

不是不能长期运行，是用对模式：

• cron → 定时任务（每次触发独立短任务）

• sessions_spawn → 后台子代理（长期运行）

• 外部进程 + 状态查询 → 后台启动，定期汇报

单次对话有 600s 限制，但 Gateway 可以 7x24 调度。

与 Manus 的区别？

竞品

Manus：云端通用 Agent，重任务执行（写代码、爬虫），网页提交任务等待结果。

OpenClaw：本地消息助手，轻量日常管理（提醒、查询、自动化），即时消息对话。

不是替代：Manus 做大事，OpenClaw 做小事，两者互补。

历史定位是什么？

先驱

参考：pi-mono（运行时）、Hubot（消息渠道）、Huginn（自动化）、Tailscale（远程访问）

独特：最早将"本地优先 + 多渠道 + Markdown 技能 + 设备协同"整合成熟

不是最早：Siri(2011)、Hubot(2013)、Huginn(2020) 在前，但 OpenClaw 是这波 LLM Agent 浪潮中最早成熟的开源本地方案之一。

🎯 核心要点

  • 架构：单一 Gateway + WebSocket + 模块化工具
  • 技能：Markdown 自然语言定义，非 JSON Schema
  • 隐私：Local-First，数据不出机，可选完全离线
  • 长期任务：用 cron 或子代理，非单次对话阻塞
  • 定位：个人消息助手，非云端重执行器
  • 成本：按需加载，小模型可跑固化工作流
