🦞 OpenClaw 架构详解

个人 AI 助手系统的控制平面设计与实现

核心理念：OpenClaw 采用「Gateway 即控制平面」的架构，所有消息渠道、客户端、设备节点都通过单一的 WebSocket 网关进行协调，实现统一的身份验证、会话管理和工具调度。

🏗️ 核心组件

1. Gateway（网关）

Gateway 是整个系统的单一控制平面，负责：

  • 渠道管理：维护与 WhatsApp、Telegram、QQ 等 20+ 消息渠道的连接
  • WebSocket 服务：默认监听 127.0.0.1:18789，所有客户端通过 WS 连接
  • 会话管理：维护活跃会话（Session）、路由消息、管理工具调用
  • Canvas Host：在同一端口提供可视化工作区

2. Agent 运行时

  • Pi Agent：基于 RPC 模式运行，支持工具流式调用（tool streaming）
  • 会话模型：
    • main：私聊会话，拥有最高权限
    • 群组会话：独立隔离上下文，可配置沙箱
  • 工具集：browser、canvas、nodes、cron、sessions 等

3. Nodes（设备节点）

macOS/iOS/Android 设备通过 WebSocket 以 role: node 接入 Gateway：

  • 暴露设备能力：camera（拍照/录像）、screen record、location、notifications
  • 通过 node.invoke 执行设备本地命令
  • 支持设备配对和设备令牌管理

📡 协议设计

WebSocket 协议帧

// 1. 连接握手 {type:"req", id:1, method:"connect", params:{deviceId:"...", token:"..."}} {type:"res", id:1, ok:true, payload:{...}} // 2. 发起对话 {type:"req", id:2, method:"agent", params:{message:"Hello"}} {type:"res", id:2, status:"accepted", runId:"run-123"} // 3. 服务端推送（流式响应） {type:"event", event:"agent", payload:{chunk:"...", runId:"run-123"}} {type:"event", event:"agent", payload:{done:true, runId:"run-123"}}

连接生命周期

1
客户端发送 connect 请求，携带设备标识和认证令牌
2
Gateway 验证设备（检查配对状态、签名验证）
3
连接建立，Gateway 推送 presence 和 tick 事件
4
客户端可发起 agent、send、status 等请求

🌐 远程访问方案

推荐方案：Tailscale 组网，无需开放公网端口

方案对比

方案 命令/配置 适用场景
Tailscale Serve gateway.tailscale.mode: "serve" 仅 tailnet 内访问（推荐）
Tailscale Funnel gateway.tailscale.mode: "funnel" 公开访问（需密码认证）
SSH 隧道 ssh -N -L 18789:127.0.0.1:18789 user@host 临时访问，无需 Tailscale

🔄 消息处理流程

1
接收：Gateway 从某个渠道（如 QQ）接收消息
2
验证：检查发件人是否在 allowFrom 白名单
3
路由：私聊 → main 会话；群组 → 独立会话
4
处理：调用 Agent 运行时，执行工具调用
5
回复：通过原渠道返回响应

🎯 设计优势

  1. 单一入口：所有渠道和客户端统一通过 Gateway，易于安全管理
  2. 渠道无关：接入新渠道只需实现 Gateway 适配器，上层无感知
  3. 设备能力无缝暴露：通过 Node 协议，手机摄像头、位置等能力可直接被 AI 调用
  4. 灵活部署：Gateway 可运行在本地、VPS、容器中，客户端通过 Tailscale/SSH 连接
  5. 会话隔离：私聊、不同群组完全隔离，支持不同权限和模型配置
注意：Gateway 设计上每主机只运行一个实例。如果需要连接多个 WhatsApp 账号，需要多台主机或多个容器实例。
